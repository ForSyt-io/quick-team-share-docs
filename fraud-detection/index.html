Existing Code: What It Does (Detail)

FileLinesWhat It DoesHow Fraud System Uses It
sharpDetection.ts 877 5-pillar scoring: CLV (30%), device fingerprint (15%), IP correlation (10%), betting patterns (20%), payment overlap (25%). Composite score: ≥70 = SHARP, ≥40 = NEW_ACCOUNT, <40 = NORMAL. Fraud risk scoring engine reads the composite sharp score as one of its inputs. No duplication — just consume the score.
collusionDetection.ts 889 Agent-punter collusion detection: cooling-off periods (24h/72h), flip frequency limits (max 3/week), collusion scoring (flip frequency + timing + P&L asymmetry + volume concentration), auto-freeze at ≥150. Fraud scoring engine reads collusion score. Agent-level fraud alerts can trigger from collusion freeze events.
filterEngine.ts ~200 Routes orders: sharp users → exchange only. Checks B-Book capacity (global/market/selection/user levels), odds & stake eligibility. Fraud blocking integrates here — blocked users get REJECT before routing decision.
alerting.ts ~150 V3Alert creation + dispatch. Alert types: COLLUSION_FREEZE, NO_NEW_RISK, SETTLEMENT_FAILURE, HEDGE_FAILED, DLQ_ENTRY, BETFAIR_DOWN. Slack + Telegram. We ADD new alert types (VELOCITY_BREACH, MULTI_ACCOUNT, COURTSIDING, MATCH_FIXING_SIGNAL, etc.) to the same pipeline. Same dispatch infrastructure.
noNewRisk.ts 643 Scope-aware bet blocking when risk thresholds are breached. Coordinates NNR state across the agent hierarchy. Fraud system's blocked entity check runs BEFORE this. If fraud blocks, NNR never evaluates.
exposureLedger.ts 582 2-tier exposure tracking. PLATFORM_EVENT_CAP for syndicate detection. Global/market/selection/user exposure limits. Used for context: large exposure + fraud signals = higher risk score weighting.
V3DeviceFingerprint schema Stores: visitorId, userAgent, platform, screenResolution, timezone, webglRenderer, ipAddress, userId. Fraud middleware writes device data here. Multi-account detection queries for same visitorId + different userId.
V3IpAccessLog schema Stores: userId, ipAddress, requestCount, firstSeen, lastSeen. Fraud middleware increments on every request. Same-IP multi-user alerts query this table.
V3Alert schema Stores: alertType, severity (P1/P2/P3), message, metadata, status (ACTIVE/RESOLVED/ACKNOWLEDGED), agentId. Fraud alerts use the SAME model. New alert types are added. Admin fraud dashboard queries this table.

New Code: What We Build on Top

ComponentTypeWhat It DoesDepends On
betContextService.ts Middleware Express middleware on every request. Extracts IP (x-forwarded-for), device FP (from request body), geolocation (MaxMind GeoIP2 lookup). Attaches to req.fraudContext. Writes V3DeviceFingerprint + V3IpAccessLog. V3DeviceFingerprint, V3IpAccessLog, MaxMind DB
fraudDetectionService.ts Service Called synchronously on every bet. Computes risk score (0-100) from 8 signals: velocity, IP multi-account, device multi-account, geo anomaly, sharp score (from sharpDetection), collusion score, slippage, cancel rate. Returns ACCEPT/MONITOR/FRICTION/DECLINE. sharpDetection.ts, collusionDetection.ts, Redis counters
fraudBlockingService.ts Service Manages BlockedEntity table. Checks fraud:blocked:* Redis cache on every request. Block types: hard_block, soft_block, warn. Supports IP, device, user, and agent blocking. BlockedEntity (new model), Redis
fraudCorrelationService.ts Background Job Runs every 60s. Detects correlated betting: same fixture+outcome within ±10s from different agent trees. IP clustering. Device clustering. Creates BetCorrelation records + V3Alert. V3DeviceFingerprint, V3IpAccessLog, V3Alert
marketSignalService.ts Background Job Taps BetfairCache stream updates. Persists LTP + volume time-series to Redis sorted sets (24h TTL). Detects steam moves (>15% in <2min), volume spikes (>3x in 30min). Creates V3Alert. BetfairCache (existing), V3Alert, Redis
LoginEvent Prisma Model Records every login: userId, IP, device FP, geo (country, city, lat/lng), wallet chain, success/failure, risk score. Enables impossible travel and new device detection. New migration
BetContext Prisma Model Records fraud metadata per bet: IP, device FP, geo, risk score, risk level, risk signals (JSON), recommendation, venue distance. Links to Order. New migration
BlockedEntity Prisma Model Stores blocked IPs, devices, users, agents. Block type (hard/soft/warn), reason, expiry, linked case. New migration
FraudCase Prisma Model Investigation cases grouping related alerts. Title, severity, status (open/investigating/resolved), assigned admin, findings, action taken. New migration
BetCorrelation Prisma Model Records correlated betting patterns. Type (same_fixture, opposing_sides, timing_cluster), correlated userIds/orderIds, correlation score. New migration
AdminAuditLog Prisma Model Logs every admin action: who, what, target, previous/new state, IP. 5-year retention. New migration
/admin/fraud/* Frontend (4 pages) Dashboard (stats + timeline), alerts (queue with filters), cases (investigation mgmt), blocked (entity management), rules (threshold config). V3Alert, FraudCase, BlockedEntity APIs

4-Layer Defense Model

Each layer adds sophistication. Layer 1 runs on day 1. Layer 4 requires months of data.

🛡

Layer 1: Static Gates

Runs on EVERY bet in the hot path. Zero historical data needed.

  • • Velocity limits (bets/hour, $/hour per user & agent tree)
  • • Blocked entity check (IP, device, user)
  • • Geolocation gate (blocked countries, VPN/TOR)
  • • Thin market stake caps
📊

Layer 2: Risk Score

Per-user composite score (0-100) checked on every bet.

  • • 8 weighted signals → single score
  • • ACCEPT / MONITOR / FRICTION / DECLINE
  • • Integrates B-Book v3 sharp score
  • • Admin-configurable thresholds
🔍

Layer 3: Async Detection

Background analysis, doesn't block bets. Generates alerts.

  • • Cross-user bet correlation (ring detection)
  • • Market manipulation signals (steam moves)
  • • Win rate anomaly (per-user, per-agent)
  • • Cancel pattern analysis
  • • Cricket event correlation (courtsiding)
🤖

Layer 4: ML / AI (Future)

Requires 2-3 months of data. Catches novel patterns.

  • • XGBoost risk score model
  • • Graph-based ring detection (Neo4j)
  • • Behavioral biometrics
  • • Anomaly detection (Z-score, clustering)

Bet Placement Flow (with Fraud Checks)

Fraud checks inject BEFORE the existing order pipeline. Target: <50ms total fraud overhead.

Login Flow (with Fraud Capture)

Every login captures context. Alerts are async and don't block the user.

Implementation Phases

Each phase delivers value independently. No phase depends on a later phase.

Phase 0 — Immediate

Security Fix

  • Fix auth.ts OR-based email lookup vulnerability
  • Require explicit confirmation for email-only account matches
Phase 1 — Tracer Bullet (Week 1-2)

IP/Device Capture + Admin Alert Queue

  • Express middleware: capture IP + user-agent on every request
  • Frontend: integrate Fingerprint.com SDK, send device FP
  • LoginEvent + BetContext Prisma models
  • FraudAlert model + alertService
  • One detection rule: same-IP multi-user alert
  • /admin/fraud page with alert list
  • WebSocket fraud:alert:new for real-time updates
Phase 2 — Risk Scoring + Blocking (Week 2-3)

Score Every Bet, Block Flagged Entities

  • fraudDetectionService.ts — risk scoring engine with configurable weights
  • BlockedEntity model — block IPs, devices, users, agents
  • Velocity limits (Redis sorted sets, sliding windows)
  • Fraud middleware injected before orderService
  • /admin/fraud/blocked — manage blocked entities
  • /admin/fraud/rules — configure thresholds
Phase 3 — Correlation + Cases (Week 3-5)

Cross-User Patterns, Investigation Tools

  • fraudCorrelationService.ts — background bet correlation
  • FraudCase model — group alerts into investigations
  • /admin/fraud/cases — case management with full user view
  • Agent-punter IP correlation alerts
  • Courtsiding detection (venue geofencing + event timing)
  • Win rate anomaly detection (post-settlement)
  • Cancel pattern tracking
Phase 4 — Market Signals (Month 2)

Exchange Data Monitoring

  • marketSignalService.ts — tap into BetfairCache stream
  • Redis time-series for LTP + volume
  • Steam move detection (>15% price change in <2min)
  • Volume spike detection (>3x normal in 30min)
  • Bet-to-price correlation (insider signal)
Phase 5 — ML / Advanced (Month 3+)

Machine Learning + Compliance

  • XGBoost/LightGBM risk model trained on historical data
  • AdminAuditLog — full admin action trail
  • Sharp user auto-detection (replaces manual-only)
  • Behavioral fingerprinting (betting profile similarity)
  • IBIA/Sportradar feed integration
  • Daily/weekly fraud reports

2026 Regulatory Landscape

Key regulations driving fraud detection requirements. Updated March 2026.

SAFE Bet Act (US)

Reintroduced 119th Congress. Max 5 deposits/24h. National self-exclusion via SAMHSA. 5-year record retention. SAR reporting to DOJ + state regulators simultaneously. Not yet passed but shapes industry standards.

GLI-GSF (NEW)

First gaming-specific cybersecurity standard. 5 modules released Oct 2025. Replaces GLI-19/GLI-33 security appendices. Covers online gaming controls, vendor controls, and technical security assessments.

UKGC Financial Checks

Now standard practice: >150 GBP/30d triggers affordability check. 97% completed "frictionlessly." AI-generated document fraud flagged as growing concern.

IBIA Mission 2030

300 suspicious alerts in 2025 (29% increase). Football (110) and tennis (74) most flagged. Global MAP monitors $300B+ turnover across 80+ sports.

Player Prop Crackdown

17+ states restrict college player props. NY actively re-examining all game-specific player props. Sen. Schatz preparing 2026 federal bill. MLB nationwide plan with FanDuel/DraftKings.

Sportradar AI

AI-based detections up 56% YoY. FIFA partnership extended to 2031. UFDS monitors 1M+ events across 70 sports. 99.5% of events clean.
